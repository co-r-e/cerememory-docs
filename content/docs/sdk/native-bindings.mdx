## Overview

Cerememory also ships native bindings that embed the Rust engine directly in your Python or Node.js process. These bindings avoid HTTP overhead and are useful for embedded or single-process workloads.

<Callout type="info">
The native bindings currently expose a smaller API surface than the HTTP SDKs. They focus on in-process store, recall, inspect, forget, and stats workflows.
</Callout>

## Python Native (`cerememory-native`)

### Build from Source

```bash
cd bindings/python-native
pip install maturin
maturin develop --release
```

The Python module name is `cerememory_native`.

### Usage

```python
from cerememory_native import Engine

engine = Engine({"working_capacity": 7, "recall_mode": "human"})

record_id = engine.store(
    "Important fact about the user",
    store="semantic",
    metadata={"source": "native"},
)

response = engine.store_full("Another fact", store="episodic")
print(response.record_id, response.initial_fidelity)

results = engine.recall("user facts", limit=5, activation_depth=2)
record = engine.get_record(record_id)
deleted = engine.forget([record_id], confirm=True)
stats = engine.stats()
```

### Exposed Methods

| Method | Notes |
|--------|-------|
| `store(text, store=None, metadata=None)` | Returns UUID string |
| `store_full(text, store=None, metadata=None)` | Python-only convenience method returning full `EncodeStoreResponse` fields |
| `recall(query, limit=10, stores=None, min_fidelity=None, recall_mode=None, reconsolidate=True, activation_depth=2)` | Returns `RecallQueryResponse` |
| `get_record(id)` | Returns a full `MemoryRecord` view |
| `forget(ids, confirm=False)` | Accepts a list of UUID strings |
| `stats()` | Returns stats plus optional evolution metrics |

## Node.js Native (`@cerememory/native`)

### Build from Source

```bash
cd bindings/typescript-native
npm install
npm run build
```

### Usage

```typescript
import { CerememoryEngine } from "@cerememory/native";

const engine = new CerememoryEngine();

const recordId = engine.store(
  "User prefers dark mode",
  "semantic",
  { source: "native" }
);

const results = engine.recall("user preferences", 5, true, 2);
const record = engine.get_record(recordId);
const deleted = engine.forget([recordId], true);
const stats = engine.stats();
```

<Callout type="warning">
The Node.js binding methods are synchronous from the caller's perspective. Do not `await` them.
</Callout>

### Exposed Methods

| Method | Notes |
|--------|-------|
| `store(text, store?, metadata?)` | Returns UUID string |
| `recall(query, limit?, reconsolidate?, activation_depth?)` | Returns JSON-shaped recall data |
| `get_record(id)` | Returns JSON-shaped record data |
| `forget(ids, confirm?)` | Deletes by UUID list |
| `stats()` | Returns JSON-shaped stats data |

## Native vs HTTP SDK

| Capability | HTTP SDKs | Native bindings |
|------------|-----------|-----------------|
| Shared server process | Yes | No |
| Auth / CORS / rate limiting | Yes | No |
| Health / readiness / metrics | Yes | No |
| Full REST API surface | Yes | No |
| In-process calls | No | Yes |
| Best for embedded single-process workloads | Sometimes | Yes |

## When to Use Native Bindings

Use native bindings when:

- You want the engine embedded directly in one process
- You do not need shared multi-client access over HTTP/gRPC
- You want to avoid transport overhead in tests or local tools

Use the HTTP server and SDKs when:

- Multiple clients share one memory store
- You need observability, auth, or remote access
- You want the full public API surface
