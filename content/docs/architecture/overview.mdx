## System Overview

Cerememory is organized as a layered system where each layer has a single responsibility. Requests flow from external clients through transport bindings, into the engine, and down to the individual memory stores.

## Layer Responsibilities

### Transport Layer

Three transport bindings expose the CMP protocol to external clients:

- **HTTP/REST** (axum) -- Primary transport on port 8420. Includes Bearer token auth, CORS, rate limiting, request tracing, and Prometheus metrics.
- **gRPC** (tonic) -- Optional transport with TLS support. Enabled by setting `grpc.port` in configuration.
- **MCP** (rmcp) -- Model Context Protocol server over stdio. Enables direct integration with MCP-compatible AI agents.

### Engine Layer

The `CerememoryEngine` is the central coordinator. It implements all CMP operations (encode, recall, lifecycle, introspect) and orchestrates cross-store interactions. It manages:

- The **hippocampal coordinator** for cross-store recall and spreading activation
- The **decay engine** for background fidelity decay (runs on a configurable interval)
- The **text index** (tantivy) and **vector index** (HNSW) for fast retrieval
- An optional **LLM provider** for embedding generation, summarization, and relation extraction

### Store Layer

Five specialized stores, each backed by redb (except working memory which is in-memory only). All stores implement the `Store` trait, providing a uniform interface for CRUD operations, text search, and fidelity updates.

### Supporting Engines

- **Decay Engine** -- Stateless, rayon-parallel computation of power-law decay across all records
- **Association Engine** -- Spreading activation over the cross-store association graph
- **Evolution Engine** -- Adaptive parameter tuning based on usage patterns

## Dependency Direction

```
cerememory-cli
├── cerememory-config       ← cerememory-engine
├── cerememory-engine       ← cerememory-core, all stores, decay, association, index
├── cerememory-transport-*  ← cerememory-engine, cerememory-core
├── cerememory-archive      ← cerememory-core
└── adapter-*              ← cerememory-core

cerememory-core (leaf crate: types, traits, protocol, errors)
```

<Callout type="info">
`cerememory-core` is the leaf dependency -- it defines all shared types, traits, and protocol structures. Every other crate depends on it, but it depends on no internal crate.
</Callout>
